\section{The Problem}

\newthought{uvicorn listens on port 8000,} but browsers expect port 80 (HTTP) and 443 (HTTPS).
You need something in front that handles the ports, serves static files, and forwards API requests.
That something is Nginx\index{Nginx}.

\section{The Frontend Stack}\index{JavaScript}\index{HTML}\index{CSS}

\subsection{Why Vanilla JavaScript}

\newthought{HTML for structure, CSS for layout, JavaScript for interactivity.} The \texttt{fetch} API
sends requests to the backend.

\marginnote{MDN Web Docs\index{MDN} (HTML/CSS/JS reference): \url{https://developer.mozilla.org/}\\
Fetch API: \url{https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API}}

\marginnote{Why not React, Vue, or Svelte? Frameworks solve problems that appear at scale, component reuse,
state management, routing across dozens of views. PixelWise has one page with one canvas and one results panel.
A framework would add a build toolchain (Node.js, npm, bundler), a learning curve orthogonal to this course,
and complexity that hides what is actually happening between browser and server. Vanilla JS keeps the HTTP
layer visible.}

\newthought{The starter code is provided.} Students need to understand it well enough to wire it up
and extend it later in the course. The key pieces:

\begin{description}
    \item[Drawing canvas] A 28$\times$28 pixel grid where users draw with the mouse.
    \item[Submit button] Extracts the pixel array from the canvas and sends a \texttt{POST /api/classify} request.
    \item[Results panel] Displays the predicted digit, confidence score, and a confidence bar.
\end{description}

\newthought{The \texttt{fetch} call} is the bridge between frontend and backend:

\begin{verbatim}
async function classifyDrawing() {
    const pixels = getPixelsFromCanvas();  // 28x28 array
    const response = await fetch("/api/classify", {
        method: "POST",
        headers: {
            "Content-Type": "application/json",
            "X-API-Key": API_KEY
        },
        body: JSON.stringify({ pixels: pixels })
    });
    const result = await response.json();
    displayResult(result);
}
\end{verbatim}

\marginnote{No build step, no \texttt{node\_modules}, no framework version to maintain. The code runs in
any browser, can be read by any developer, and deploys as static files served directly by Nginx. Students
who later build larger applications will recognise when a framework becomes necessary, and why.}

\section{Nginx: Three Roles in One}\index{Nginx}

\newthought{Nginx serves three roles}\marginnote{Nginx\index{Nginx}: \url{https://nginx.org/}\\
Nginx Docs: \url{https://nginx.org/en/docs/}} in the PixelWise stack:

\begin{description}
    \item[Static file server] Serves HTML, CSS, and JavaScript files far more efficiently than Python.
    \item[Reverse proxy]\index{reverse proxy} Forwards \texttt{/api/} requests to \texttt{localhost:8000} (uvicorn).
    The browser never talks to uvicorn directly.
    \item[SSL terminator]\index{TLS}\index{SSL} Handles HTTPS encryption. The backend receives plain HTTP; encryption
    is Nginx's job.
\end{description}

\subsection{Nginx Config Anatomy}

\newthought{Nginx configuration} is structured in nested blocks.
A \texttt{server} block declares the listening port and host name, then
groups one or more \texttt{location} blocks that match request paths.
A \texttt{location /} block can serve static files from a \texttt{root}
directory, and a \texttt{location /api/} block can hand traffic off to
uvicorn with \texttt{proxy\_pass http://localhost:8000/}, where the
trailing slash strips the \texttt{/api} prefix before the request reaches
the backend.
The exercise below has the full file PixelWise needs.

\marginnote{The \texttt{proxy\_pass} directive is the reverse proxy: any request to \texttt{/api/classify}
is forwarded to \texttt{http://localhost:8000/classify}. The browser sees \texttt{https://192.168.56.11/api/classify};
uvicorn sees \texttt{http://localhost:8000/classify}. The trailing slash matters, it strips the \texttt{/api}
prefix.}

\section{Exercise: Install and Configure Nginx}\index{exercises}

\newthought{Install Nginx} on \texttt{prod}:

\begin{verbatim}
sudo apt install -y nginx
\end{verbatim}

\newthought{Author the site configuration} on \texttt{dev} at
\texttt{deploy/nginx/pixelwise.conf} in the repository, alongside the
systemd unit from Block~5. The config is part of how the app runs, so
it lives next to the code.

\begin{verbatim}
# deploy/nginx/pixelwise.conf
server {
    listen 80;
    server_name 192.168.56.11;

    # Serve static files
    location / {
        root /opt/pixelwise/frontend;
        index index.html;
    }

    # Proxy API requests to uvicorn
    location /api/ {
        proxy_pass http://localhost:8000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
\end{verbatim}

Commit, push from \texttt{dev}, and pull on \texttt{prod}. Then install
the site by copying the committed file into Nginx's search path,
enable it, and restart:

\begin{verbatim}
sudo cp /opt/pixelwise/deploy/nginx/pixelwise.conf \
    /etc/nginx/sites-available/pixelwise
sudo ln -s /etc/nginx/sites-available/pixelwise \
    /etc/nginx/sites-enabled/
sudo rm /etc/nginx/sites-enabled/default
sudo nginx -t          # test config syntax
sudo systemctl restart nginx
\end{verbatim}

\marginnote{\texttt{nginx -t} parses the configuration without applying it. Run it before every restart;
a syntax error caught here is cheaper than a daemon that refuses to start in production.}

\newthought{Verify the reverse proxy.} Open a browser on the host machine and navigate to
\texttt{http://192.168.56.11}. You should see the PixelWise frontend. Draw a digit and submit, the
request goes through Nginx to uvicorn and back. The browser sees a single origin; uvicorn never appears
in any URL.

\newthought{Teach \texttt{setup-server.sh} to install the site.}\index{setup-server.sh}
The five commands you just typed belong in
\texttt{scripts/setup-server.sh} so a fresh VM picks up the site
without manual steps. Open the file in nano on \texttt{dev}:

\begin{verbatim}
nano setup-server.sh
\end{verbatim}

\marginnote{\newthought{Same prod-only guard.}\index{idempotence}
The \texttt{id produser} check from Block~5 keeps Nginx out of
\texttt{dev}, where there is no public traffic to serve. The
\texttt{ln -sf} flag overwrites an existing symlink instead of
failing, so the stanza is safe to rerun.}

\begin{verbatim}
# Install the Nginx site on prod
if [ -f deploy/nginx/pixelwise.conf ] && \
   command -v nginx >/dev/null 2>&1 && \
   id produser >/dev/null 2>&1; then
    sudo cp deploy/nginx/pixelwise.conf \
        /etc/nginx/sites-available/pixelwise
    sudo ln -sf /etc/nginx/sites-available/pixelwise \
        /etc/nginx/sites-enabled/pixelwise
    sudo rm -f /etc/nginx/sites-enabled/default
    sudo nginx -t && sudo systemctl reload nginx
fi
\end{verbatim}

Add \texttt{nginx} to the \texttt{apt install} line so the binary is
present before the stanza runs. Commit, push, and verify on
\texttt{prod} that \texttt{bash setup-server.sh} reaches the same
state on a second run without errors.

\section{CORS: Cross-Origin Resource Sharing}\index{CORS}

\newthought{CORS is a browser security policy}\marginnote{CORS explained:
\url{https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS}} that blocks requests from one origin to another. When your
frontend is served on port 443 and the API runs on port 8000, the browser considers them different origins
and blocks the request.

\newthought{During development,} CORS causes pain. The browser's console shows a cryptic error:
\texttt{Access to fetch has been blocked by CORS policy}. The fix is to register the FastAPI
\texttt{CORSMiddleware} with an explicit \texttt{allow\_origins} list naming the frontend origin,
plus the methods and headers the frontend will send.
The exercise below wires it up.

\marginnote{Never use \texttt{allow\_origins=["*"]} in production, it allows any website to call your API.
Configure allowed origins explicitly.}

\newthought{With Nginx as a reverse proxy,} CORS often resolves itself: the frontend and API share the
same origin (\texttt{https://192.168.56.11}), so the browser does not block the request. But students must
understand CORS because they will hit it in every deployment that separates frontend and backend.

\newpage
\section{Exercise: Debug CORS}\index{exercises}

\newthought{If the frontend cannot reach the API,} check the browser console for CORS errors. Open the
developer tools, switch to the console tab, and reload the page; a blocked request prints a message
beginning with \texttt{Access to fetch has been blocked by CORS policy}.

\newthought{Add the CORS middleware} to FastAPI with the correct allowed origin:

\begin{verbatim}
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://192.168.56.11"],
    allow_methods=["*"],
    allow_headers=["*"],
)
\end{verbatim}

Restart uvicorn and reload the page. The console should now be silent for the previously blocked request.

\marginnote{This is a rite of passage. Every developer who has shipped a frontend separated from its
backend has read this error message at least once. Knowing the cause makes the cure a one-line edit
instead of an afternoon of bafflement.}

\section{URL Design}\index{URL design}

\newthought{Clean, readable URLs} matter:

\begin{description}
    \item[Good] \texttt{/results?digit=7\&model=v2}
    \item[Bad] \texttt{/getResults.php?action=list\&d=7}
\end{description}

\newthought{Query parameters} serve filtering and configuration. Storing session IDs in URLs
(\texttt{/classify?session=abc123}) lets you track a user's session without cookies, link to specific
results, or pass configuration to the frontend. The URL is state, treat it deliberately.

\section{HTTPS and TLS}\index{HTTPS}\index{TLS}

\newthought{Without HTTPS, API keys travel in plaintext.} Anyone on the network can read them.
HTTPS encrypts the entire channel between browser and server.

\begin{description}
    \item[Self-signed certificate] Fine for development. The browser shows a warning, ignore it.
    \item[Let's Encrypt]\index{Let's Encrypt} Free, automated certificates for production (\url{https://letsencrypt.org/}).
    Renewal is automatic via \texttt{certbot}\index{certbot} (\url{https://certbot.eff.org/}).
\end{description}

\newthought{Generating a self-signed certificate} is a one-liner with
\texttt{openssl req -x509}, which produces a key file under
\texttt{/etc/ssl/private/} and a certificate file under
\texttt{/etc/ssl/certs/} that Nginx can then point at.
The exercise below has the full command.

\subsection{Security Headers}\index{security headers}

\newthought{Nginx can add security headers} to every response, one place, all responses covered:

\begin{verbatim}
add_header X-Frame-Options "DENY";
add_header X-Content-Type-Options "nosniff";
add_header Content-Security-Policy "default-src 'self'";
\end{verbatim}

\newpage
\section{Exercise: Enable HTTPS and Security Headers}\index{exercises}

\newthought{Generate a self-signed certificate} on the server:

\begin{verbatim}
sudo openssl req -x509 -nodes -days 365 \
    -newkey rsa:2048 \
    -keyout /etc/ssl/private/pixelwise.key \
    -out /etc/ssl/certs/pixelwise.crt
\end{verbatim}

\newthought{Update \texttt{deploy/nginx/pixelwise.conf}} on \texttt{dev}
to listen on port 443 with the certificate. Replace the
\texttt{listen 80;} line with:

\begin{verbatim}
listen 443 ssl;
ssl_certificate /etc/ssl/certs/pixelwise.crt;
ssl_certificate_key /etc/ssl/private/pixelwise.key;
\end{verbatim}

Commit, push, pull on \texttt{prod}, then rerun
\texttt{setup-server.sh} to reinstall the site and reload Nginx. Open
\texttt{https://192.168.56.11} in the browser and accept the warning.

\newthought{Add security headers} to the Nginx config: \texttt{X-Frame-Options}, \texttt{X-Content-Type-Options},
and \texttt{Content-Security-Policy}. Restart and verify they appear in the response headers:

\begin{verbatim}
curl -I https://192.168.56.11 -k
\end{verbatim}

\marginnote{The \texttt{-k} flag tells \texttt{curl} to accept the self-signed certificate without
complaint. In production with a Let's Encrypt certificate, you would not need it; the warning
disappears and the headers are the same.}

\section{Optional: \texttt{ufw}}\index{ufw}\index{firewall}

\marginnote{\texttt{ufw}\index{ufw} (Uncomplicated Firewall) is Ubuntu's firewall tool. On managed production
servers, the firewall is typically preconfigured to allow only ports 22 (SSH), 80 (HTTP), 443 (HTTPS).
On the classroom VMs this is a formality (host-only network), but students should know the concept exists.}

\newthought{The firewall is the outer skin of the server.} On a host-only network there is no public
internet to defend against, so the lecture VMs run without one. On a real production host the rule of
thumb is: deny by default, allow exactly the ports the service needs. \texttt{ufw} expresses this in
a few commands and stays out of the way once configured.
