\section{What a Model File Actually Is}\index{pickle}\index{joblib}

\newthought{A \texttt{.pkl} file is a build artefact,} not source code. You distribute it, you do not commit it.
Just as a compiled binary is produced by a compiler, a model file is produced by a training script. The training
script is source code; the model file is output.

\marginnote{The course provides \texttt{models/digit\_classifier\_v1.pkl}, a LogisticRegression\index{LogisticRegression}
pipeline trained on MNIST digits 1 to 9 (9 classes, ${\sim}$54\,000 training samples, public domain).
Class 0 is intentionally held back; students can add it later without collecting any new data.}

\newthought{The training script} (\texttt{train.py}) is provided as supplementary lecture material. Students who
want to understand how the artefact was produced can run it: MNIST downloads automatically via
\texttt{sklearn.datasets.fetch\_openml}, training takes ${\sim}$30 seconds on CPU. Proper training practices
(learning rate schedules, regularisation, early stopping, GPU considerations) would double the length of this course.

\marginnote{For further reading on training practices:\\
Karpathy, \emph{A Recipe for Training Neural Networks}:
\url{https://karpathy.github.io/2019/04/25/recipe/}\\
Google, \emph{Rules of Machine Learning}:
\url{https://developers.google.com/machine-learning/guides/rules-of-ml}}

\subsection{The Deserialisation Risk}

\newthought{Loading a \texttt{.pkl} from an untrusted source can execute arbitrary code.}
\texttt{pickle}\index{pickle} and \texttt{joblib}\index{joblib} deserialise objects by reconstructing them—including
any code embedded in the file. Only load artefacts you produced or received from a trusted source.

\section{Honouring the Model Contract}\index{train/serve skew}

\newthought{Every model was trained on inputs in a specific format.} Violate that format and it predicts
silently wrong—no error, just garbage output. This is called \textbf{train/serve skew}\index{train/serve skew}
and it is the most common production ML bug.

\newthought{The MNIST model expects:}
\begin{itemize}
    \item A 28$\times$28 greyscale image
    \item Pixel values normalised to $[0, 1]$
    \item Flattened to a 784-element vector
\end{itemize}

\subsection{Binarization}\index{binarization}

\newthought{The canvas in the browser produces uint8 values} (0–255). Converting to binary (0/1)
reduces payload size, removes noise, and matches training-time preprocessing exactly. Binarized digits
are cleaner inputs—show the difference visually: a raw digit with anti-aliased grey edges versus the
same digit binarized to sharp black-and-white.

\subsection{The sklearn Pipeline}\index{sklearn!Pipeline}

\newthought{The sklearn Pipeline} bundles preprocessing and model into one serialised object. Load it once
and the preprocessing is guaranteed to match training. No separate normalisation step to forget.

\marginnote{scikit-learn\index{scikit-learn}: \url{https://scikit-learn.org/}\\
joblib\index{joblib}: \url{https://joblib.readthedocs.io/}\\
NumPy\index{NumPy}: \url{https://numpy.org/}}

\section{Designing the Inference Interface}\index{inference interface}

\newthought{Before writing any code, design the contract.} Start with the class roster, then the function signatures.

\subsection{The Class Dictionary}

\newthought{A named constant} makes the valid output set explicit and independent of the model file:

\begin{verbatim}
CLASSES = ["1", "2", "3", "4", "5", "6", "7", "8", "9"]
# must match model.classes_ — verified at startup
# note: "0" is intentionally absent in v1
\end{verbatim}

This list belongs in the code, not only in the model. The frontend, the DB schema, and the API docs all
reference it. If the model is swapped for one trained on different classes, the assertion fires loudly
at startup rather than returning silently wrong labels.

\subsection{Batch-First Design}\index{batch inference}

\newthought{Scalability question:} what happens when 50 students submit a drawing at the same second?
If the server processes each request one at a time, the model is called 50 times with a $(1, 784)$ array each.
If it batches them, the model is called once with a $(50, 784)$ array—sklearn's \texttt{predict\_proba}
is natively vectorised, so the cost is nearly identical.

\newthought{Design around batch;} single-user is a special case:

\begin{description}
    \item[\texttt{classify\_batch} (core)] Input: $(N, 28, 28)$ uint8 array. Output: list of $N$ result dicts.
    Called by the Block~5 API.
    \item[\texttt{classify} (wrapper)] Input: $(28, 28)$ uint8 array. Output: single result dict.
    Used by \texttt{predict.py} for smoke testing.
\end{description}

\subsection{The Function Signatures}

\begin{verbatim}
def classify_batch(images: np.ndarray) -> list[dict]:
    """
    Args:
        images: (N, 28, 28) uint8 numpy array, values 0-255
    Returns:
        List of N dicts, each:
        {
            "prediction": str,       # element of CLASSES
            "confidence": float,     # probability of top class
            "scores": dict[str, float]  # prob per class
        }
    Raises:
        ValueError: if shape != (N, 28, 28)
    """
    if images.ndim != 3 or images.shape[1:] != (28, 28):
        raise ValueError(
            f"Expected (N,28,28), got {images.shape}")
    arr = (images > 128).astype(float).reshape(
        len(images), -1)                    # binarize + flatten
    probs = _pipeline.predict_proba(arr)    # (N, 9)
    return [
        {"prediction": CLASSES[p.argmax()],
         "confidence": float(p.max()),
         "scores": dict(zip(CLASSES, p.tolist()))}
        for p in probs
    ]

def classify(image: np.ndarray) -> dict:
    """Convenience wrapper — single image."""
    return classify_batch(image[np.newaxis])[0]
\end{verbatim}

\marginnote{The API in Block~5 always calls \texttt{classify\_batch}, even for a single drawing—it
passes a $(1, 28, 28)$ array. When a request queue is added later, no signature changes are needed.}

\newthought{Load \texttt{\_pipeline} once at module level}—deserialising on every request kills throughput.
The model stays in memory for the lifetime of the process.

\section{The Model Card}\index{model card}\index{MODELCARD.md}

\newthought{A model card is the documentation artefact} that links a model to its provenance and limitations.
It is an established standard (Google, HuggingFace\marginnote{Model Cards (Google):
\url{https://modelcards.withgoogle.com/}\\
HuggingFace Model Cards: \url{https://huggingface.co/docs/hub/model-cards}}) and contains:

\begin{description}
    \item[Training data] MNIST digits 1–9, 9 classes, ${\sim}$54k train / ${\sim}$9k test samples, public domain. Class 0 withheld.
    \item[Capabilities] Predict handwritten digits 1–9. Expected accuracy ${\sim}$92\% (LogisticRegression baseline).
    \item[Known failures] 6/9 confusion, 3/8 confusion, messy or rotated digits.
    \item[Benchmarks] Accuracy per class on MNIST test set (provided alongside the model).
    \item[Intended use] Drawn digits from 28$\times$28 canvas. Out-of-scope: photographed digits, non-digit characters.
\end{description}

When \texttt{v2} is deployed via the Block~8 feature flag, the model card gets updated—traceability across versions.

\begin{enumerate}
    \item \textbf{Download and inspect the model:} Place \texttt{digit\_classifier\_v1.pkl} (provided)
    in \texttt{models/}. Inspect it in a Python shell:

    import joblib
    model = joblib.load("models/digit_classifier_v1.pkl")
    print(type(model))

    \end{verbatim}
    What type is it? What classes does it predict?

    \item \textbf{Experiment with binarization:} Load a sample MNIST digit, display the raw pixel array
    versus the binarized version. Observe that the digit structure is preserved and noise is eliminated:
    \begin{verbatim}
    from sklearn.datasets import fetch_openml
    import numpy as np
    X, y = fetch_openml("mnist_784", version=1,
        return_X_y=True, as_frame=False)
    sample = X[0].reshape(28, 28).astype(np.uint8)
    binary = (sample > 128).astype(float)
    print(sample[:5, :5])    # raw values
    print(binary[:5, :5])    # 0s and 1s
    \end{verbatim}

    \item \textbf{Write \texttt{app/classifier.py}:} Implement the module with the class constant,
    module-level model loading, the startup assertion, \texttt{classify\_batch}, and \texttt{classify}:
    \begin{verbatim}
    CLASSES = ["1","2","3","4","5","6","7","8","9"]

    _pipeline = joblib.load(os.getenv("MODEL_PATH"))
    assert list(_pipeline.classes_) == CLASSES, \
        f"Model/CLASSES mismatch: {_pipeline.classes_}"

    The assertion is the key teaching moment: it fires at startup. A swapped model with different
    classes is caught the moment the service starts, not when a user gets a wrong digit.

    import numpy as np

    # Load 5 test samples (provided or via fetch_openml)
    images = ...  # (5, 28, 28) uint8
    results = classify_batch(images)
    for r, true_label in zip(results, true_labels):
        print(f"Predicted: {r['prediction']} "
              f"(conf: {r['confidence']:.2f}), "
              f"True: {true_label}")

    calls \texttt{classify\_batch} with a $(1, 28, 28)$ array.

    \begin{verbatim}

\subsection{Optional Challenge: Add the Missing 0}\index{model versioning}

\newthought{The digit class ``0'' was intentionally excluded from v1.} To produce v2: add \texttt{"0"} to
\texttt{CLASSES}, include MNIST's class-0 samples in the training split, re-run \texttt{train.py}, save as
\texttt{digit\_classifier\_v2.pkl}. The assertion will break the moment v2 is loaded against v1's
\texttt{CLASSES}—fix it by updating the constant. Update \texttt{MODELCARD.md}: document that v2 covers
all 10 digits and note any accuracy change.

\marginnote{This challenge connects directly to Block~8: v2 is deployed via the feature flag, and Block~9's
analytics compare v1 vs v2 performance. No data collection needed—MNIST has the 0s already.}

\subsection{Teaser: Could We Have Spotted the Missing Class?}

\newthought{Think about it before reading the hints:}
\begin{itemize}
    \item \textbf{Class distribution in the DB:} after a few hundred predictions, a missing class is invisible—the
    model never predicts it, users never see an error, the DB just never accumulates any 0s. Would you have noticed?
    \item \textbf{Confidence as a signal:} when the model sees an input it was not trained on, it is often less certain.
    A systematic drop in average confidence or a spike in low-confidence predictions could indicate out-of-distribution
    inputs. It is a weak signal, but it is a runtime signal that requires no labelled data.
    \item \textbf{Confidence thresholds:} return \texttt{null} when \texttt{confidence < threshold}. The volume of
    rejected predictions is itself a monitoring signal.
\end{itemize}

These questions do not have quick answers. Systematic detection of silent model failures—uncertainty estimation,
distribution shift, missing classes—is the domain of \textbf{MLOps}\index{MLOps}. This surface is intentionally
shallow here; it is covered in depth in dedicated courses.

\section{Security Thread}

\newthought{Never load a \texttt{.pkl} from an untrusted source}—\texttt{joblib}/\texttt{pickle}
deserialisation executes arbitrary code. \texttt{MODEL\_PATH} comes from \texttt{.env}, never
hardcoded—so swapping the model in production requires only an env var change, not a code change.
Training data and model artefacts are not committed to Git.
