\section{The Why}

\marginnote{\newthought{PixelWise} now lives on GitHub and on both machines.
But the repository contains only a README and a \texttt{.gitignore}.
Before we can write application code, we need a reproducible way to install
everything the project depends on, from system packages to Python libraries,
so that \texttt{dev} and \texttt{prod} stay in sync without human memory.}

Block~2 left us with a PixelWise repository on GitHub and a server that has Git and Python installed.
But those packages were installed by hand, one \texttt{sudo apt install} at a time.
If the VM is rebuilt, or a new team member joins, nobody remembers what to install or in which order.

\newthought{``Works on my machine''} is the most dangerous sentence in software engineering.
Your colleague clones the repo, runs the code, and gets an \texttt{ImportError}. You installed
\texttt{scikit-learn} months ago and forgot. They did not. The code is correct; the environment is wrong.

\newthought{Dependency management} solves this at two levels: system packages and Python packages.
For system packages, we capture every \texttt{apt install} in a shell script called \texttt{setup-server.sh}\index{setup-server.sh}
so any fresh VM can be provisioned identically. For Python packages, we declare what the project needs
in a \texttt{requirements.txt}, pin the exact versions, and anyone can reconstruct the identical environment
in two commands.

% -----------------------------------------------------------------------------
\subsection{Hands On Experience}

\newthought{Consider this scenario:} you hand your PixelWise repo to a classmate.
They clone it, type \texttt{python3 train.py}, and immediately hit \texttt{ModuleNotFoundError: No module named 'sklearn'}.
You tell them to install scikit-learn. They do, but get version 2.0 while you wrote the code against 1.4.
The training script crashes with a different error. Now you are debugging version mismatches instead of building features.

\marginnote{\newthought{The two-command promise.}
A well-managed project lets any new developer go from clone to running code
with exactly two commands: create the virtual environment and install from a pinned requirements file.
No guessing, no Slack messages asking ``which version of numpy do I need?''}

\newthought{This third block} introduces dependency management and project structure for PixelWise.
By the end you will have
\begin{itemize}
    \item captured system-level setup in a reproducible shell script,
    \item created and activated a Python virtual environment,
    \item installed packages with pip and pinned them in \texttt{requirements.txt},
    \item separated secrets from code using \texttt{.env} and \texttt{.env.example},
    \item scaffolded the PixelWise project directory,
    \item and audited your dependencies for known security vulnerabilities.
\end{itemize}

\section{The Setup Script}\index{setup-server.sh}

\newthought{On the server VM,} you installed packages manually in Block~1 and Block~2.
Now capture that work in a script so any fresh VM can be provisioned identically:

\begin{verbatim}
#!/bin/bash
# setup-server.sh: provision a fresh server VM
set -euo pipefail

sudo apt update
sudo apt install -y git python3 python3-pip \
    python3-venv curl
\end{verbatim}

\marginnote{The \texttt{set -euo pipefail} line makes the script fail immediately on any error
(\texttt{-e}), on undefined variables (\texttt{-u}), and on pipe failures (\texttt{-o pipefail}).
This prevents silent failures that leave the server in a half-configured state.}

\newthought{Mark it executable} and run it:

\begin{verbatim}
chmod +x setup-server.sh
./setup-server.sh
\end{verbatim}

\newthought{This is version control paying for itself.} The script is committed to Git,
so when a new team member joins or a VM is rebuilt, \texttt{bash setup-server.sh}
reproduces the exact same environment. No Slack messages, no guessing.

\marginnote{As the project grows, this script may split into multiple files or graduate to
a configuration management tool like Ansible\index{Ansible}.
The principle stays the same: every setup step lives in version control.\\
Ansible: \url{https://docs.ansible.com/}\\
GNU Make: \url{https://www.gnu.org/software/make/}}

\newthought{System packages vs. Python packages.} The setup script handles system-level dependencies:
Git, Python itself, system libraries that compiled Python packages need.
Python-level dependencies live in \texttt{requirements.txt} and are managed by pip inside a virtual environment.
The two layers are complementary: the setup script prepares the ground, the virtual environment grows the project.

\section{Virtual Environments}\index{virtualenv}\index{venv}

\newthought{A virtual environment} is an isolated Python installation. Each project gets its own
set of packages, independent of every other project and the system Python.

\begin{verbatim}
python3 -m venv .venv
source .venv/bin/activate
\end{verbatim}

\marginnote{The \texttt{.venv/} directory contains a full Python installation: interpreter, \texttt{pip},
and all installed packages. It is machine-specific and large, often 50 to 200\,MB.
It must be in \texttt{.gitignore}: never commit it. Anyone can recreate it from \texttt{requirements.txt}.}

\newthought{While the environment is active,} \texttt{python} and \texttt{pip} point to the local copies.
Install a package and it stays inside \texttt{.venv/}, with no system-wide pollution. Deactivate with \texttt{deactivate}.

\newthought{Verify the switch:}

\begin{verbatim}
which python
# /home/user/pixelwise/.venv/bin/python

pip list
# Package    Version
# ---------- -------
# pip        24.0
# setuptools 69.5.1
\end{verbatim}

A fresh environment starts empty. Every package you install from this point is tracked and intentional.

\marginnote{The data science ecosystem often uses \texttt{conda}\index{conda} instead of \texttt{venv}.
Conda manages both Python packages and system-level libraries like CUDA or MKL, which makes it popular
for GPU workflows. For web projects, \texttt{venv} is the lighter and more standard choice.\\
Conda: \url{https://docs.conda.io/}}

\newthought{The rule is simple:} every project gets its own \texttt{.venv}. Project A needs
\texttt{numpy==1.24}; Project B needs \texttt{numpy==2.0}. Without isolation, one of them breaks.
With virtual environments, they coexist peacefully on the same machine.

\bigskip
\section{Managing Packages with pip}\index{pip}\index{requirements.txt}

\newthought{pip is the package installer} for Python. The core cycle:

\begin{verbatim}
pip install scikit-learn joblib python-dotenv
pip freeze > requirements.txt
pip install -r requirements.txt
\end{verbatim}

\begin{itemize}
    \item \texttt{pip install} downloads and installs packages from PyPI\index{PyPI} (the Python Package Index): \url{https://pypi.org/}.
    \item \texttt{pip freeze} lists every installed package with its exact version.
    \item \texttt{pip install -r} installs everything listed in a requirements file, the reproducibility step.
\end{itemize}

% -----------------------------------------------------------------------------
\subsection{Anatomy of \texttt{requirements.txt}}

\newthought{One package per line,} pinned to an exact version:

\begin{verbatim}
scikit-learn==1.4.0
joblib==1.3.2
python-dotenv==1.0.1
\end{verbatim}

\marginnote{Version specifiers: \texttt{==} pins exactly, \texttt{>=} sets a floor, \texttt{\~{}=} allows
patch updates (\texttt{\~{}=1.4.0} means \texttt{>=1.4.0, <1.5.0}). For maximum reproducibility, pin with \texttt{==}.\\
Semantic versioning: \url{https://semver.org}}

\marginnote{\newthought{Security auditing.} Your dependencies have their own dependencies,
and any of them can harbor known vulnerabilities. \texttt{pip-audit}\index{pip audit}\index{CVE}
checks every installed package against the CVE database:
\texttt{pip install pip-audit \&\& pip-audit}.
Run it after every \texttt{pip freeze}. In Block~8 we automate this with a cron job
so vulnerabilities surface without human memory.\\
\texttt{pip-audit}: \url{https://github.com/pypa/pip-audit}\\
Dependabot: \url{https://docs.github.com/en/code-security/dependabot}}

\section{Configuration: \texttt{.env} and \texttt{python-dotenv}}\index{.env}\index{python-dotenv}\index{12-factor app}

\newthought{Separating configuration from code} is a foundational principle. The 12-factor app methodology
states: store config in the environment, never in the source code.

\marginnote{The Twelve-Factor App: \url{https://12factor.net/}\\
python-dotenv: \url{https://github.com/theskumar/python-dotenv}}

\newthought{A \texttt{.env} file} holds key-value pairs that your application reads at startup:

\begin{verbatim}
SECRET_API_KEY=my-actual-secret-key-here
DEBUG=true
\end{verbatim}

In Python, \texttt{python-dotenv}\index{python-dotenv} loads these into environment variables:

\begin{verbatim}
from dotenv import load_dotenv
import os

load_dotenv()
api_key = os.getenv("SECRET_API_KEY")
debug = os.getenv("DEBUG", "false").lower() == "true"
\end{verbatim}

% -----------------------------------------------------------------------------
\subsection{The \texttt{.env.example} Pattern}\index{.env.example}

\newthought{The \texttt{.env} file contains real secrets,} so it must never be committed.
The \texttt{.env.example} file is the contract: it shows which variables exist, what they are for,
and provides safe placeholder values:

\begin{verbatim}
# API key callers must send in the X-API-Key header
# (a secret, never a real value here)
SECRET_API_KEY=replace-me

# Debug mode: when true, FastAPI shows /docs and full
# error tracebacks. Never true in production.
DEBUG=true
\end{verbatim}

\marginnote{\texttt{.env.example} is committed to Git and public. \texttt{.env} is in \texttt{.gitignore}
and stays local. The contrast is immediate: one is the template, the other is the secret.
This pattern grows block by block, every new service adds its variables to \texttt{.env.example}.}

\newthought{Students create their actual \texttt{.env}} by copying \texttt{.env.example} and replacing the placeholders
with real values.

\section{The Container Alternative}\index{Docker}

\marginnote{Docker\index{Docker} and Docker Compose solve the ``reproducible environment'' problem at the
OS level rather than the Python level. This course teaches the virtualenv path to keep the OS layer visible
and learnable. Docker is the natural next step after this course.\\
Docker: \url{https://www.docker.com/}\\
Docker Docs: \url{https://docs.docker.com/get-started/}}

\newthought{Virtual environments isolate Python packages.} Docker containers isolate the entire operating system:
libraries, system packages, Python version, everything. Where \texttt{requirements.txt} says ``install these
Python packages,'' a \texttt{Dockerfile}\index{Dockerfile} says ``start from this Linux image, install these system packages,
then install these Python packages.''

\begin{verbatim}
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "main.py"]
\end{verbatim}

The result is a self-contained unit that runs identically on any machine
with Docker installed. We do not use Docker in this course, but you should know it exists and why teams adopt it.

\section{Optional: When \texttt{requirements.txt} Starts to Hurt}\index{pyproject.toml}

\marginnote{Tools like \texttt{uv}\index{uv} and \texttt{poetry}\index{poetry} use this model and are fast
becoming the standard. Once the \texttt{requirements.txt} workflow is second nature, this is the upgrade.\\
\url{https://packaging.python.org/en/latest/}}

\newthought{\texttt{pip freeze > requirements.txt}} captures everything: your direct dependencies and all their
transitive dependencies\index{transitive dependencies}, flattened into one list. Six months later you want
to upgrade one package: which of those 47 lines do you actually own? \texttt{pyproject.toml} solves this
by separating \emph{what you depend on}, your direct deps loosely pinned, from \emph{what gets installed},
a generated lock file\index{lock file} with the full resolved graph. Upgrades become surgical: bump one line, regenerate
the lock, done.

\begin{verbatim}
[project]
name = "pixelwise"
version = "0.1.0"
dependencies = [
    "scikit-learn>=1.4",
    "joblib>=1.3",
    "python-dotenv>=1.0",
]
\end{verbatim}

\newthought{Compare this to \texttt{requirements.txt}:} the project file lists only what you chose to install,
with flexible version bounds. The tool resolves the full dependency graph and writes a lock file
that pins every transitive dependency. You commit both files; the lock file is your reproducibility guarantee.

\section{Exercise: The Setup Script}\index{exercises}

\marginnote{\newthought{Exercises} are for practice and reinforcing concepts.
Work through them yourself first, break things, discuss, this is not a time trial.
If your VM ends up in a state you cannot recover, restore your own \texttt{B2-complete} VM snapshot
or follow the tag catch-up below.
At the end of the session, take a snapshot named \texttt{B3-complete}, your safety net for the next block.}

\newthought{Catching up via the reference tag.}\index{Git!tag}\index{recovery}
If you joined late, missed the previous block, or your VM is in a state you cannot recover,
you do not need to redo every prior exercise. The course maintains a public reference
repository at \url{https://github.com/schutera/pixelwise} with a tag at the end of every block.
Restore the distributed \texttt{B1-complete} VM image, complete the Block~2 Git identity and SSH
key setup so the VM has your credentials, then clone the reference repository at the
\texttt{v0.1} tag, which marks the end of Block~2:

\begin{verbatim}
cd ~
git clone --branch v0.1 https://github.com/schutera/pixelwise.git
cd pixelwise
git switch -c main
git remote set-url origin git@github.com:<you>/pixelwise.git
\end{verbatim}

You are now at the state where Block~2 ended, ready to start Block~3.

\marginnote{\newthought{Why a tag, not a VM snapshot?}\index{Git!tag}
The reference repository carries a tag at the end of every block: \texttt{v0.1} for Block~2,
\texttt{v0.2} for Block~3, and so on, all visible at
\url{https://github.com/schutera/pixelwise/tags}.
Each tag points at a specific commit, so checking out a tag rewinds the working tree to the
exact state the block ended on. The tag carries no credentials, only the project history.
You then repoint \texttt{origin} at your own GitHub repository so future pushes go to your
account, not the reference one. From this block onwards, \texttt{setup-server.sh} and
\texttt{requirements.txt} reproduce the system and Python environments on top, so a fresh VM goes
from empty to running by combining the \texttt{B1-complete} image, the reference tag, and the scripts.}

\newthought{Write \texttt{setup-server.sh}.} On the \texttt{dev} VM, inside the PixelWise repo,
create a shell script that captures the system packages you installed manually in earlier blocks.
Add \texttt{set -euo pipefail} at the top so the script fails loudly on any error.

\begin{verbatim}
#!/bin/bash
set -euo pipefail

sudo apt update
sudo apt install -y git python3 python3-pip \
    python3-venv curl
\end{verbatim}

\newthought{Commit it.} This is the first file that makes provisioning reproducible:

\begin{verbatim}
git add setup-server.sh
git commit -m "Add server provisioning script"
git push
\end{verbatim}

\section{Security Thread}

\newthought{\texttt{pip audit}\index{pip audit}} catches known CVEs in dependencies; run it regularly and before
every deploy. \texttt{.env.example} makes the secrets contract explicit without exposing values.
Never commit \texttt{.env}. The \texttt{.venv/} folder must be in \texttt{.gitignore}: it is large,
machine-specific, and reconstructable from \texttt{requirements.txt} alone.

\section{Self-Reflection and Recap}

\newthought{Reflection questions:}
\begin{itemize}
    \item Why do we use a virtual environment instead of installing packages system-wide?
    \item What is the difference between \texttt{.env} and \texttt{.env.example}? Why do we need both?
    \item What does \texttt{pip freeze} capture that you did not explicitly install?
    \item If a colleague clones your repo, what commands do they run to get a working environment?
    \item What is the risk of committing \texttt{.env} to Git, even briefly?
\end{itemize}

\newthought{Recap:}
\begin{itemize}
    \item \texttt{setup-server.sh} makes system-level provisioning reproducible and auditable.
    \item Virtual environments isolate Python packages per project; never install globally.
    \item \texttt{requirements.txt} pins every dependency so both machines run identical code.
    \item \texttt{.env.example} documents secrets; \texttt{.env} holds the real values and never enters Git.
    \item \texttt{pip audit} catches known vulnerabilities in your dependency tree.
\end{itemize}
